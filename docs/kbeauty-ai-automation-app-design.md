# K뷰티 AI 직원 자동화 운영앱 설계도

## 1. 한 줄 정의

`처음_열기.html`을 단순 안내 문서가 아니라, 상국님이 돈을 쓰기 전에 AI 직원들이 상품·링크·콘텐츠·위험·성과를 검증하고 다음 일을 자동 배정하는 로컬 운영앱으로 확장한다.

## 2. 최종 목표

| 목표 | 설명 |
|---|---|
| 사장실 대시보드 | 상국님은 첫 화면에서 현재 단계, 오늘 할 일, 승인 필요 작업만 본다 |
| AI 직원 자동 배정 | 상품 조사, 콘텐츠 작성, 위험 검수, 성과 분석을 직원별 작업으로 나눈다 |
| 작업 큐 | 대기, 진행 중, 완료, 승인 필요 작업을 JSON으로 저장한다 |
| 승인 게이트 | 로그인, 결제, 발행, 개인정보 입력은 상국님 승인 없이는 실행하지 않는다 |
| 자동 보고 | 와이프 공유용 보고서, 사장님 보고서, 다음 우선순위 1~3순위를 자동 생성한다 |
| 확장 가능 구조 | 이후 StyleKorean, YesStyle, Shopee Affiliate, SNS 발행 자동화를 붙일 수 있게 한다 |

## 3. 현재 상태와 목표 상태

| 구분 | 현재 | 목표 |
|---|---|---|
| 실행 방식 | `처음_열기.html` 직접 열기 | `npm start`로 로컬 서버 실행 |
| 저장 방식 | 브라우저 localStorage | `data/*.json` 파일 저장 |
| AI 직원 | 화면상 역할 안내 | `config/kbeauty-workers.json` 직원 설정 |
| 작업 기록 | 없음 또는 브라우저 내부 | `data/kbeauty-command-log.json` |
| 자동화 | 버튼 중심 반자동 | API가 작업 접수, 분류, 결과 저장 |
| 승인 관리 | 문구 안내 | 승인 필요 작업을 별도 상태로 분리 |

## 5. AI 직원 10명 설계

| AI 직원 | 역할 | 자동화 가능 | 승인 필요 |
|---|---|---:|---:|
| 👑 AI 팀장 | 명령 접수, 작업 배정, 다음 우선순위 작성 | ✅ | 일부 |
| 🔎 상품 검증원 | COSRX, Torriden, 3CE 후보 상품 정리 | ✅ | ❌ |
| 🏷️ 제휴 링크 관리자 | 링크 입력 여부, 링크 자리, 고지 문구 확인 | ✅ | 링크 발급은 상국님 |
| ✍️ 콘텐츠 작가 | 블로그 글, 쇼츠 대본, 제목 후보 작성 | ✅ | 발행은 상국님 |
| 🛡️ 리스크 검수원 | 과장 표현, 치료·완치 표현, 민감정보 경고 | ✅ | ❌ |
| 📊 성과 분석가 | CTR, CVR, 구매 여부, 확장 판단 | ✅ | ❌ |
| 💬 공유문 작성자 | 와이프 공유용 메시지, 보고서 요약 작성 | ✅ | 전송은 상국님 |
| 📦 상품 확장 전략가 | 다음 상품 후보와 테스트 순서 추천 | ✅ | 재고/구매는 상국님 |
| 🔐 승인 관리자 | 로그인, 결제, 발행, 개인정보 입력 분리 | ✅ | ✅ |
| ⚙️ 운영 관리자 | 서버 상태, 파일 저장, 작업 로그 점검 | ✅ | 일부 |

## 6. 자동화 가능/불가능 경계

| 영역 | 자동화 판단 | 이유 |
|---|---|---|
| 상품 후보 정리 | 가능 | 공개 정보와 입력 데이터를 바탕으로 후보표 생성 |
| 콘텐츠 초안 작성 | 가능 | 블로그, 쇼츠, 제목, 설명문 생성 가능 |
| 링크 자리 검수 | 가능 | 원고에 링크와 제휴 고지 존재 여부 확인 |
| 성과 계산 | 가능 | 조회수, 클릭수, 구매수, 커미션 기반 계산 |
| 다음 행동 추천 | 가능 | 규칙 기반으로 대기/수정/확장 판단 |
| StyleKorean 가입 | 반자동 | 외부 사이트 계정 생성은 상국님 입력 필요 |
| PayPal 연결 | 수동 | 금융/정산 정보는 상국님 직접 입력 |
| 콘텐츠 발행 | 승인 후 | 외부 전송 행위라 최종 승인 필요 |
| 광고비 집행 | 승인 후 | 돈이 나가는 작업 |
| 재고 구매 | 승인 후 | 돈과 리스크가 생기는 작업 |

## 7. API 설계 초안

| API | 기능 | 저장 위치 |
|---|---|---|
| `GET /api/status` | 운영앱 상태, 현재 단계, 완료율 조회 | 계산값 |
| `GET /api/workers` | AI 직원 목록과 현재 업무 조회 | `config/kbeauty-workers.json` |
| `POST /api/command` | 상국님 명령 접수 | `data/kbeauty-command-log.json` |
| `GET /api/tasks` | 작업 큐 조회 | `data/kbeauty-tasks.json` |
| `POST /api/tasks/run-safe` | 안전 작업 실행 | `data/kbeauty-tasks.json` |
| `POST /api/campaigns` | 성과 숫자 저장 | `data/kbeauty-campaigns.json` |
| `GET /api/reports/today` | 오늘 보고서 생성 | `data/kbeauty-reports.json` |
| `POST /api/approval/request` | 승인 필요 작업 등록 | `data/kbeauty-tasks.json` |

## 9. MVP 구현 순서

| 순서 | 작업 | 결과물 |
|---:|---|---|
| 1 | `package.json`, `server.js` 생성 | 로컬 서버 실행 가능 |
| 2 | `config/kbeauty-workers.json` 생성 | AI 직원 10명 설정 |
| 3 | `data` JSON 저장소 생성 | 작업/성과/보고서 저장 가능 |
| 4 | 기존 `처음_열기.html`을 서버에서 제공 | 현재 UI 유지 |
| 5 | 명령 접수 API 추가 | 사장실 명령 로그 저장 |
| 6 | 작업 큐 API 추가 | 대기/진행/완료/승인필요 분리 |
| 7 | 성과 저장 API 연결 | localStorage 의존 줄이기 |
| 8 | 오늘 보고서 API 연결 | 와이프 공유문/사장님 보고서 자동 생성 |

## 10. 첫 MVP에서 하지 않을 것

| 제외 항목 | 이유 |
|---|---|
| 실제 쇼피 자동 상품 등록 | 계정/API/정책 확인 전 위험 |
| PayPal 자동 입력 | 민감 금융 정보 |
| 자동 발행 | 외부 전송이라 승인 체계 먼저 필요 |
| 유료 광고 자동 집행 | 비용 리스크 |
| 재고 자동 구매 | 사업 리스크 |

## 11. 완료 기준

| 기준 | 확인 방법 |
|---|---|
| 로컬 서버 실행 | `npm start` 후 브라우저 접속 |
| AI 직원 목록 표시 | `/api/workers` 응답 확인 |
| 명령 접수 가능 | `/api/command` 저장 확인 |
| 작업 큐 표시 | 대시보드에서 대기/완료/승인 필요 확인 |
| 보고서 생성 | 오늘 보고서 JSON과 화면 출력 확인 |
| 승인 게이트 동작 | 발행/결제/로그인 작업이 자동 실행되지 않고 승인 필요로 분리 |

## 12. 결론

첫 구현은 “완전 자동화”보다 “안전한 자동화 운영실”이 맞다.
AI 직원들은 조사, 작성, 검수, 분석, 보고를 자동으로 하고, 상국님은 로그인·결제·발행·구매처럼 책임이 생기는 일만 승인한다.
